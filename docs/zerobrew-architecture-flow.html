Zerobrew Architecture and Flow

Current-state architecture for the Rust workspace after the cleanup, isolated linking, main-only list, upgrade metadata refresh, stricter dependency graph, cask safety, cache validation, and zb run behavior changes.

Workspace: zb_cli, zb_io, zb_core CLI binary: zb Run shim: zbx DB schema: v3 Default macOS prefix: /opt/zerobrew

1. System Overview

Zerobrew is split into a small CLI crate, a core model crate, and an IO-heavy crate that owns networking, extraction, install state, linking, cask handling, source builds, and cleanup.

Command Router

CommandMain pathImportant current behavior
zb install, zb iplan then executeRequested inputs become main packages. Dependencies are installed as dependency-only.

Runtime Paths

PathRole
$ZEROBREW_ROOT/cacheAPI response cache and SHA-addressed downloaded blobs.
$ZEROBREW_ROOT/storeExtracted content-addressed store entries keyed by bottle SHA or source key.
$ZEROBREW_ROOT/Cellar/name/versionMaterialized keg tree. Formula-private files live here.

~/Applications/*.appDefault cask app target on macOS. Existing app bundles are overwritten by default.

2. Install Planning Flow

Planning is strict. A required runtime dependency must resolve and must have either a usable bottle or a source build plan. Tapped formula dependencies resolve same-tap first for unqualified names, then fall back to core only when same-tap metadata is missing. 

flowchart TD
  Start["Install input names"] --> Normalize["CLI normalizes formula or cask names"]
  Normalize --> Fetch["fetch_all_formulas(names)"]
  Fetch --> IsTap{"Formula is tapped?
owner/repo/name"}
  IsTap -- "no" --> CoreDeps["Keep dependencies as listed"]
  IsTap -- "yes" --> DepLoop["For each unqualified dependency"]
  DepLoop --> TrySameTap["Try owner/repo/dependency"]
  TrySameTap --> SameTapFound{"Found same-tap formula?"}
  SameTapFound -- "yes" --> UseSameTap["Store resolved dependency as owner/repo/dependency"]
  SameTapFound -- "404 only" --> UseCore["Fall back to core dependency name"]
  SameTapFound -- "network or parse error" --> AbortTap["Abort planning"]
  CoreDeps --> QueueDeps["Queue unresolved dependency names"]
  UseSameTap --> QueueDeps
  UseCore --> QueueDeps
  QueueDeps --> More{"More queued deps?"}
  More -- "yes" --> Fetch
  More -- "no" --> Closure["resolve_closure roots plus deps"]
  Closure --> Missing{"Any dependency node missing?"}
  Missing -- "yes" --> AbortMissing["Abort with MissingFormula"]

Requested metadata: A formula is main only when it appears in the user install arguments. If a dependency is installed explicitly later, DB upsert promotes it to requested and never demotes it during later dependency work.
Cleanup dependency safety: Cleanup protects dependencies based on the persisted installed graph recorded at install time, not whatever upstream metadata says today.

3. Formula Install Execution

Bottles download in parallel and are processed as soon as each completes. Source builds run after bottle work because they need installed dependency paths. 

sequenceDiagram
  autonumber
  actor U as User
  participant C as CLI command
  participant I as Installer
  participant A as ApiClient
  participant D as Downloader
  participant S as Store
  participant K as Cellar
  participant L as Linker
  participant B as Database

  U->>C: zb install jq
  C->>I: plan requested names
  I->>A: fetch formula and dependencies
  A-->>I: resolved formula map
  I->>I: resolve_closure dependencies first
  I-->>C: install plan
  C->>I: execute plan with linking
  I->>I: acquire install.lock
  I->>D: stream bottle downloads
  D-->>I: download result as each bottle finishes
  I->>S: verify and extract store entry
  S-->>I: extracted store path
  I->>K: materialize keg
  I->>L: unlink previous keg links on reinstall
  I->>B: record install and dependency graph
  B-->>I: installed state committed
  I->>L: update opt link
  I->>L: link shared or isolated files
  L-->>I: link outcome
  I->>B: record linked files
  I->>K: remove old keg after successful upgrade
  I-->>C: execute result
  C-->>U: installed count and warnings
Retry path: If store extraction detects corruption, the bad blob is removed and the downloader retries up to the corruption retry limit.
Upgrade lifecycle: Previous public and isolated links are removed before relinking the new keg, so binaries or completions removed upstream do not stay behind as stale commands.

What Linking Means

Cellar is truth: The package files live in Cellar/name/version.
Links expose files: Prefix files are symlinks pointing back into the active keg.
bin is public: Executables are exposed in prefix/bin whenever their names do not collide.
libexec is private: Vendored runtimes like Python site-packages are not recursively shared into prefix/libexec.

DB Semantics

Field/tableMeaning
requestedMain package marker. Default zb list filters on this field.
deps_recordedWhether the installed formula has direct runtime dependency edges recorded.
installed_dependenciesDirect resolved dependency names at install time. Cleanup walks this graph.
store_refsReference counts for extracted store entries. GC removes entries with refcount <= 0 and no installed keg reference.
keg_filesAbsolute linked paths so uninstall can remove shared, isolated, and external cask artifacts.

6. Cleanup Flow

Cleanup removes dependency-only formulae that are no longer reachable from requested installed formulae. It does not auto-remove casks, and it aborts if a requested root lacks recorded dependency metadata. 

flowchart TD
  Start["zb cleanup or uninstall --cleanup"] --> Load["Load all installed_kegs"]
  Load --> Roots["Protected roots = requested=true formulae
excluding casks"]
  Roots --> Walk["Walk installed_dependencies graph"]
  Walk --> Missing{"Protected node has deps_recorded=0?"}
  Missing -- "yes" --> Abort["Abort before removing anything
ask user to reinstall or reset"]
  Missing -- "no" --> Protect["Protected set = roots plus reachable deps"]
  Protect --> Candidate["Candidate = installed formula
requested=false, not cask, not protected"]
  Candidate --> Dry{"--dry-run?"}
  Dry -- "yes" --> Print["Print packages that would be removed
do not uninstall, do not GC"]
  Dry -- "no" --> Remove["Uninstall each candidate
links, DB, keg"]
  Remove --> GC["Run store GC"]
  GC --> Done["Print removed dependencies and removed store entries"]

What Happens When

SituationOutcome
Dependency remains after root uninstallYes. Normal uninstall removes only the target. Run zb cleanup or zb uninstall target --cleanup to remove unused dependency-only formulae.
Keg-only dependency has executablesNon-conflicting executable names are linked into prefix/bin; dependency-only keg-only warnings stay quiet unless an executable-name conflict occurs.
Two packages ship the same support filePrivate trees like libexec/site-packages are not shared into the prefix, so duplicate support files do not create noisy conflicts.
Two packages ship the same executableThe existing owner stays untouched. The new package still installs, and only the executable-name conflict is reported.
API returns bad JSON with HTTP 200The body is rejected before it can poison the API cache.
Cached bottle blob is corruptedThe blob is rehashed before reuse, deleted on mismatch, and downloaded again.
